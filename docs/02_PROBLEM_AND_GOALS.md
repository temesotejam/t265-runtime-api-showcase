# Problem and goals

## 解こうとした問題

T265を実験やロボット制御で使うとき、単に「SDKのサンプルを動かす」だけでは足りない場面があります。

よくある課題は次のようなものです。

- Linux / Ubuntu のバージョン差でSDK環境が崩れやすい。
- T265が runtime mode で見えているのか bootloader mode で見えているのか分かりにくい。
- pose / IMU / fisheye の流れを用途別に整理したい。
- 古いsampleを処理するより、制御周期ごとに最新値だけ読みたい。
- ログ保存では逆にsampleを順番に取りたい。
- 複数T265を使うと、USB index だけでは左右・前後を固定しにくい。
- 画像payloadを扱うと、bufferの所有権が曖昧になりやすい。
- 公開しようとすると firmware や実機シリアルが混ざる。

## 設計目標

このプロジェクトでは、次のような目標で設計しました。

### 1. Linux/libusb中心にする

T265を扱う最小構成を把握するため、なるべく下の層でデバイスの状態を見る方針にしました。

狙いは、公式SDKを否定することではありません。むしろ、SDKが何を抽象化してくれているのかを理解し、自分の用途に必要な最小部分を見極めるためです。

### 2. runtime modeを中心にする

T265にはUSB上の見え方が変わる状態があります。

- runtime mode: 通常利用状態
- bootloader / Movidius mode: firmware load が必要になる状態

API本体は runtime mode を対象にし、bootloaderからruntimeへ戻す部分は補助機能として切り分けました。

### 3. latest と queue を分ける

これはかなり重要な設計判断です。

ロボット制御では、古いposeを大量に処理するより、今のposeが欲しいことが多いです。逆に、ログ解析では過去のsampleを順番に処理したい。

この2つを同じAPIで無理に扱うと、どちらにも中途半端になります。

そこで、次のように分けました。

| 用途 | API |
|---|---|
| 制御・現在値取得 | latest state API |
| ログ・時系列解析 | queue / multi-queue API |

### 4. callback内は軽くする

データ受信側のcallbackで画像保存や重い解析を行うと、受信処理そのものが詰まりやすくなります。

そのため、callbackでは基本的に軽い処理だけ行います。

- latest state を更新する
- queue に push する
- 統計値を更新する

重い処理は、pop側やworker thread側へ逃がす方針です。

### 5. 複数台運用を最初から意識する

T265を2台以上使う場合、USBのindexだけでは扱いにくくなります。抜き差しで順番が変わる可能性があるためです。

そのため、serial number と role file の考え方を取り入れました。

例としては、left / right / front / rear のような役割名を serial に紐づける方式です。

### 6. 公開境界を設計に含める

今回のプロジェクトでは、公開できるものと公開しない方がよいものを早い段階で分ける必要がありました。

公開しない方がよいもの:

- firmware blob
- 実機シリアル番号
- USB生ダンプ
- 個体依存ログ
- ビルド済みバイナリ

公開するもの:

- 設計思想
- 調査観点
- API分類
- 構成図
- 検証で見るべき指標

この切り分けにより、成果は示しつつ、危ないファイルを外せます。
