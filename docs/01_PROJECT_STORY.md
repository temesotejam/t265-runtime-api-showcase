# Project story

## 何を作ったのか

Intel RealSense T265 を Linux 上で扱うための、実験的な runtime API を作りました。

目的は、単にサンプルを動かすことではなく、T265をロボット制御や検証用途で扱いやすい形に整理することでした。特に、公式SDKに強く依存しすぎると環境差で詰まりやすい場面があるため、Linux / USB / libusb という下の層から T265 の runtime 状態を観察し、C API として使いやすい形にまとめる方針を取りました。

## 最初に考えたこと

最初に考えたのは、「T265をどう抽象化するか」です。

T265はカメラですが、今回の用途では単なる画像入力装置として見るより、次のように分けて見る方が整理しやすかったです。

- pose を出すデバイス
- IMU sample を出すデバイス
- fisheye metadata / image を出すデバイス
- USB状態が変わるデバイス
- 起動状態によって見え方が変わるデバイス

この見方をすると、やるべきことがかなり明確になります。

1. デバイスが今どの状態で見えているか確認する。
2. runtime状態のデバイスを開く。
3. streamを設定する。
4. データを受け取る。
5. 用途別に最新値・queue・同期へ振り分ける。
6. 長時間動作で詰まりや取りこぼしを観察する。

## 作ってみて分かったこと

重要だったのは、APIを1つにまとめすぎないことでした。

ロボット制御では、過去のsampleを全部処理するより、現在の姿勢・速度・IMUをすぐ読みたい場合があります。一方、ログ保存や解析では、順番にsampleを処理できないと困ります。

そのため、設計上は次のように分けました。

| 目的 | APIの考え方 |
|---|---|
| 制御 | latest state API |
| ログ・解析 | queue API |
| 複数種類のsample処理 | multi-queue API |
| 画像と姿勢の対応付け | syncer |
| 最短確認 | simple reader |
| 複数台運用 | serial / role based open |

## 何を成果として見せるべきか

このプロジェクトの成果は、コード量そのものより、下の層を観察して使いやすいAPI境界まで持ち上げたことです。

特に見せたいポイントは次の通りです。

- USB上の状態を見て、runtime / bootloader を切り分けたこと。
- 制御用途と解析用途でAPIを分けたこと。
- callback内で重い処理をしない設計にしたこと。
- queueの詰まりを観察できるようにしたこと。
- 画像payloadの所有権を意識したこと。
- firmwareや実機シリアルを公開しない境界を作ったこと。

## この公開版の立ち位置

全実装は非公開で管理し、この公開版では設計思想と調査観点だけを公開します。

これは「隠している」というより、firmware・実機固有情報・ライセンス境界を混ぜずに成果を見せるための切り分けです。むしろ、この切り分け自体も成果の一部だと考えています。
