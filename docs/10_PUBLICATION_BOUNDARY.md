# Publication boundary

## なぜ境界を決める必要があるか

今回のプロジェクトは、普通の自作ライブラリより公開時に注意が必要です。

理由は、次のものが混ざりやすいからです。

- firmware binary
- 実機シリアル番号
- USB生ダンプ
- 個体依存ログ
- ビルド済みバイナリ
- SDKや他者著作物との境界が分かりにくい情報

そのため、公開用には「設計資料だけ」に切り分けるのが安全です。

## 公開してよいもの

| 種類 | 理由 |
|---|---|
| 設計思想 | 自分の判断・整理なので成果として見せやすい |
| アーキテクチャ図 | 全体像を伝えやすい |
| API分類 | 何をどう分けたかが分かる |
| 検証観点 | 他の人にも役立つ |
| 公開境界の説明 | 安全に配慮していることが伝わる |
| 抽象的な調査メモ | 方向性を示せる |

## 公開しない方がよいもの

| 種類 | 理由 |
|---|---|
| firmware blob | 再配布権限が不明確になりやすい |
| firmwareを含むobject file | blobを間接的に含む可能性がある |
| 実機シリアル番号 | 個体情報であり公開不要 |
| USB生ダンプ | 個体情報・解析詳細・不要な情報を含む可能性がある |
| ビルド済みバイナリ | ソースから再現できるもの。混入物も確認しにくい |
| private logs | 環境依存・個体情報が混ざりやすい |
| 全実装コード | firmware周りや境界確認が済むまでは非公開が安全 |

## この公開版の方針

この公開版は、次の原則で作っています。

1. コードは載せない。
2. firmwareは載せない。
3. 実機固有情報は載せない。
4. 実装の再現に直結する低レベル詳細は載せすぎない。
5. 何を調べればよいか、どう考えればよいかを中心に書く。

## READMEに明記するべきこと

公開する場合、READMEには次を明記しておくと安全です。

- 非公式プロジェクトであること。
- 実装コード全体は非公開であること。
- firmware binaryは含めていないこと。
- 実機固有設定は含めていないこと。
- このリポジトリは設計・調査記録であること。

## Public / Private の2段構成

おすすめは次の構成です。

| Repository | Visibility | 内容 |
|---|---|---|
| private repo | Private | 全コード、firmware、実機設定、詳細ログ |
| showcase docs repo | Public | 設計思想、調査観点、構成図、検証方針 |

この分け方なら、成果は見せられますが、危ないファイルは出しません。

## 自分用メモ

今回のようなハードウェア寄りプロジェクトでは、「全部公開すること」が正義ではありません。

むしろ、公開境界を設計できている方が、技術的にも権利的にも落ち着いた成果になります。
